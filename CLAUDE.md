# CLAUDE.md — komunify

This file provides guidance to Claude Code when working in this monorepo.

## Project Overview

**komunify** is a Web3 dApp on Stellar (Soroban) — a unified community subscription
platform with automated on-chain revenue splitting. See `README.md` for product scope.

Packages:
- `contracts/` — Soroban (Rust) smart contracts. Cargo workspace, not an npm package.
- `packages/contract-client/` — **Generated** TypeScript bindings for the contract (the "ABI"). Consumed by web + api.
- `packages/api/` — Hono backend API (port 3001)
- `packages/web/` — Next.js frontend (port 3000) with Freighter wallet integration
- `packages/shared/` — Shared Zod schemas, TS types, and Stellar network config/constants

## Stack

- **Blockchain**: Stellar / Soroban smart contracts (Rust, `soroban-sdk`), Stellar CLI 27+
- **Contract client**: `@stellar/stellar-sdk` v14 bindings generated by `stellar contract bindings typescript`
- **Backend**: Hono, TypeScript, Zod, Prisma (postgres), better-auth
- **Frontend**: Next.js 15 App Router, TanStack Query, Tailwind CSS, shadcn/ui, better-auth client, Freighter wallet (`@stellar/freighter-api`)
- **Shared**: Zod schemas, inferred TS types, Stellar network config
- **Package Manager**: bun
- **Monorepo**: bun workspaces (`packages/*`)

## Development Commands

```bash
# Run everything
bun dev                         # Start all packages in parallel

# Individual packages
bun --filter api dev            # Backend only
bun --filter web dev            # Frontend only

# Build
bun build                       # Build all packages
bun --filter shared build       # Build shared first if types changed

# Quality

bun lint                        # Lint all packages

```

> `contract-client` is built to `dist/` (committed) — rebuild it after regenerating bindings.

## Smart Contract (Soroban)

The `notes` contract (`contracts/contracts/notes/`) is a minimal per-owner note manager:
`add_note`, `update_note`, `delete_note`, `get_note`, `list_notes`. Each note belongs to an
owner address; only the owner can mutate their notes (`require_auth`). Reads are open.

> This is the current MVP contract. The broader subscription / revenue-split contract from
> `README.md` is future scope — start here.

Use the **Makefile** (from repo root) for the full workflow:

```bash
make setup     # one-time: add wasm target + create/fund a testnet identity ("deployer")
make test      # cargo unit tests
make build     # compile to Wasm (contracts/target/wasm32v1-none/release/notes.wasm)
make bindings  # regenerate packages/contract-client from the Wasm + rebuild dist
make deploy    # deploy to testnet, writes the id to .contract-id
make invoke   ARGS="add_note --owner <G...> --title Hi --content Yo"
make simulate ARGS="list_notes --owner <G...>"
```

Then set the printed id in `packages/web/.env.local` as `NEXT_PUBLIC_NOTES_CONTRACT_ID`.

Equivalent bun scripts exist for CI: `bun contract:test`, `bun contract:build`, `bun contract:bindings`.

Prereqs: Stellar CLI (`brew install stellar-cli`), Rust, and the Wasm target
(`rustup target add wasm32v1-none`, done by `make setup`).

## Wallet (Freighter) + calling the contract

Frontend integration lives in `packages/web`:
- `providers/wallet-provider.tsx` — `<WalletProvider>` + `useWallet()` (connect/disconnect, address, network)
- `lib/stellar.ts` — runtime network config from `NEXT_PUBLIC_*` env + `@komunify/shared` defaults
- `lib/notes-client.ts` — `getNotesClient(publicKey?)` returns a contract client that signs via Freighter
- `components/wallet/connect-wallet-button.tsx` — connect button (design-system styled)

The notes feature follows the `services/` domain pattern (types / queries / service / hook),
but the service calls the Soroban contract via the bindings instead of the REST `ApiClient`:
- `services/notes/notes.service.ts` — reads simulate over RPC (no wallet); writes sign via Freighter
- `services/notes/notes.hook.ts` — `useNotes`, `useCreateNote`, `useUpdateNote`, `useDeleteNote`
- `components/notes/notes-app.tsx` + `app/page.tsx` — the UI, styled per `DESIGN.md` (SPLIT v4)

## Database

- **Provider**: PostgreSQL via Prisma
- Schema: `packages/api/prisma/schema.prisma`

```bash
bun --filter api prisma:generate   # Regenerate Prisma client
bun --filter api prisma:migrate    # Run migrations
bun --filter api prisma:studio     # Open Prisma Studio
```

## Authentication

Powered by [better-auth](https://better-auth.com).

- Sessions stored in database, sent as HTTP-only cookies
- Auth routes handled at `/api/auth/**` on the backend
- Frontend uses `lib/auth-client.ts` — no manual token management
- Required env vars: `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`

See `packages/api/CLAUDE.md` for protecting routes.
See `packages/web/CLAUDE.md` for frontend usage.
